タスクをユーザーに依存させる

タスクをユーザーに依存させる

現在の作りでは、作成されたタスクを誰もが閲覧できてしまいますが、できればユーザー毎に表示される内容を取得したいところです。 それを実現させるため、タスクは自分で登録したものしか閲覧できないようにプログラムを修正していきましょう。

マイグレーションによるテーブル情報操作

tasksテーブルを作成する際、下記のようにしてテーブルを作成したことを思い出してください。

class CreateTasksTable extends Migration
{
    /**
     * Run the migrations.
     *
     * @return void
     */
    public function up()
    {
        Schema::create('tasks', function (Blueprint $table) {
            $table->increments('id');
            $table->string('name', 255);
            $table->string('detail', 255);
            $table->datetime('deadline', 0)->nullable();
            $table->timestamps();
        });
    }

このtasksテーブルにuser_idカラムを追加し、外部キー制約を設定していきます。
Laravelのマイグレーションとは時系列にデータベースの構造管理をするための仕組みなので、 一度実行されたマイグレーション用のファイルはロールバックされない限り２度と実行はされません。 そのため、新しく更新用のマイグレーションファイルを作成して変更を行う必要があります。

まずは、下記のコマンドで更新用のマイグレーションファイルを作成してください。

php artisan make:migration add_user_id_on_tasks

作成されたファイルを確認してみましょう。

use Illuminate\Support\Facades\Schema;

use Illuminate\Database\Migrations\Migration;

class AddUserIdOnTasks extends Migration
{
    /**

     *
     * @return void
     */
    public function up()
    {
        //
    }

    /**
     * Reverse the migrations.
     *
     * @return void
     */
    public function down()
    {
        //
    }
}

これまでと違ってupメソッドに処理が書かれてないのに気がついたと思います。
これは、Laravelに「xxxテーブルへの操作だよ」という考慮をさせるためのトリガーが無いためです。 Laravelは、make:migration を行う際の最後の引数である名前を読み取り処理内容の自動入力を行ってくれます。

php artisan make:migration add_user_id_on_tasks
                           ~~~~~~~~~~~~~~~~~~~~

次に、作成されたdatabases/migrations/yyyy_mm_dd_add_user_id_on_tasks.phpを一旦削除して、 再度下記のコマンドを実行してみましょう。

php artisan make:migration add_user_id_to_tasks_table

そしてまた、作成されたファイルを確認してみてください。

今度は「どのテーブルに」という処理内容が予め書かれた状態でファイルが追加されたと思います。
これは名前の最後に tasks_table が追加されているため、「tasksテーブルに関する処理か」と 自動で判別してくれているためです。

ここで記入されている処理は、「tasksテーブルに対する処理だよ」というのを意味する部分になります。 そのため、この { } の中に何をしたいのかを書いていけば良いのです。

        Schema::table('tasks', function (Blueprint $table) {
            // カラム変更などに関する処理を記述する
        });

早速ここに、user_idカラムを追加し、外部キー制約を設定するための処理を書いていくのですが、 その前にターミナルまたはコマンドプロンプトから下記のコマンドを実行しておいてください。

composer require doctrine/dbal

doctrine/dbalとは、現在のカラムの状態を確認し、指定されたカラムに対する修正を行うSQLクエリを生成するための機能ですが、 上記のコマンドを実行することでこの機能をLaravelに取り込んでいます。

doctrine/dbalの取り込みが完了したら、ようやくadd_user_id_to_tasks_table.phpの内容の修正をしてみましょう。

<?php

use Illuminate\Support\Facades\Schema;
use Illuminate\Database\Schema\Blueprint;
use Illuminate\Database\Migrations\Migration;

class AddUserIdToTasksTable extends Migration
{
    /**
     * Run the migrations.
     *
     * @return void
     */
    public function up()
    {
        Schema::table('tasks', function (Blueprint $table) {
            // 追加：user_id カラムを初期値 1 として追加
            $table->unsignedInteger('user_id')->default(1);
            // 追加：user_id カラムに対して users.id への外部キーを on Delete Cascade で設定
            $table->foreign('user_id')->references('id')->on('users')->onDelete('cascade');
        });
    }

    /**
     * Reverse the migrations.
     *
     * @return void
     */
    public function down()
    {
        Schema::table('tasks', function (Blueprint $table) {
            // 追加：設定した外部キーを削除
            $table->dropForeign('tasks_user_id_foreign');
            // 追加：user_id カラムを削除
            $table->dropColumn('user_id');
        });
    }
}

ここでは、追加したuser_idカラムとusersテーブルのidを、外部キーでつないで、onDelete Cascadeを設定しています。
onDelete Cascadeとは、外部キーで紐付けされた親に当たるレコードが削除された場合、関連する子のレコードもすべて 削除するというオプションになります。

【例】
tasks.user_idの値が 1 となっているレコードが複数あった場合、users.idの値が 1 のレコードを削除すると、それらがすべて削除される。

修正が完了したらマイグレーションを実行します。

php artisan migrate

下記のように表示されたら実行は完了です。

Migrating: 2020_03_18_002341_add_user_id_to_tasks_table
Migrated:  2020_03_18_002341_add_user_id_to_tasks_table (0.09 seconds)

早速データベースを開いて内容を確認してみましょう。
tasksテーブルにuser_idというカラムが追加されているのを確認できたのではないでしょうか？ これで、データベースとしてのタスクとユーザーの紐付けは完了です。

登録タスク内容の変更

データベースとしての紐付けができても、取得するデータがすべてのタスクでは、これまでとあまり変わりがありません。
次の修正として、ユーザーが自分のタスクのみを参照できるよう、登録と取得の処理も修正していきましょう。

まずは、TodoControllerの登録処理で「誰が」そのタスクを登録したのかという情報を書き込むように修正します。


namespace App\Http\Controllers;

use Illuminate\Http\Request;
use App\Http\Requests\StoreTaskPost;
use App\Task;
// 追加：Authファサード
use Illuminate\Support\Facades\Auth;

class TodoController extends Controller
{

    ...（省略）...

    public function store (StoreTaskPost $request)
    {
        $input = $request->all();
        unset($input['_token']);

        // 追加：ログインしているユーザーのIDをuser_idに設定
        $input['user_id'] = Auth::id();

        $tasks = new Task;
        $tasks->fill($input);
        $tasks->save();

        return redirect(route('task.index'));
    }

次に、モデルから登録処理可能なカラムの定義も修正しておきます。

<?php

namespace App;

use Illuminate\Database\Eloquent\Model;

class Task extends Model
{
    // 修正：user_id を追加
    protected $fillable = ['name', 'detail', 'deadline', 'user_id'];

修正ができたら、新しいユーザーを登録してそのユーザーでログインした状態からタスクの追加を実行してみましょう。
タスクの追加ができたら、データベースに追加されたレコードのuser_idがログイン中のユーザーのIDになっていることを確認してください。

タスク一覧の表示内容の変更

最後に、表示に関しても修正をしておきましょう。
現在はすべてのタスクが取得され一覧として表示されます。 これを、ログインしているユーザーのデータのみ取得できるように修正していきます。

Tasksモデルを開いてget_taskメソッドを下記のように修正しましょう。

    public function get_task($id)
    {
        $ret = $this->where('user_id', $id)->get();

        return $ret;
    }

次に、呼び出し元のTodoControllerコントローラーも修正します。

    public function index ()
    {
        $task = new Task;
        $list = $task->get_task(Auth::id());

        return view('index', [
               'list'  => $list
        ]);
    }

修正ができたら、タスク一覧を表示してみてください。
ログインしているユーザーが登録したタスクのみが表示されるようになっているはずです。
